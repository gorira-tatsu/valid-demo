# 01. 共通仕様

## データモデル

### 投稿
- `id`: 投稿識別子
- `title`: タイトル
- `body`: 本文
- `authorName`: 投稿者名
- `editKey`: 投稿編集用キー
- `createdAt`: 作成日時
- `updatedAt`: 更新日時
- `deletedAt`: 論理削除日時。未削除時は `null`

### コメント
- `id`: コメント識別子
- `postId`: 紐づく投稿 ID
- `body`: コメント本文
- `authorName`: コメント投稿者名
- `createdAt`: 作成日時
- `deletedAt`: 論理削除日時。未削除時は `null`

## 識別子・時刻の共通ルール
- `id` は一意で、同一種別内で再利用しない
- `createdAt` は作成成功時に一度だけ確定する
- `updatedAt` は未更新時は `null` とし、投稿更新成功時のみ更新する
- `updatedAt` は `createdAt` 以上であること
- 一覧の既定順は `createdAt` の降順とし、同一時刻の場合は `id` の降順で安定化する
- コメントの既定順は `createdAt` の昇順とし、同一時刻の場合は `id` の昇順で安定化する
- 一覧再取得時は、同一データ集合に対して同一順序を返す

## 共通バリデーション

### 投稿
- タイトル: 必須、1 文字以上 100 文字以下
- 本文: 必須、1 文字以上 5,000 文字以下
- 投稿者名: 任意、未入力時は `名無しさん`
- 編集用キー: 必須、4 文字以上 32 文字以下

### コメント
- 本文: 必須、1 文字以上 1,000 文字以下
- 投稿者名: 任意、未入力時は `名無しさん`

## 共通画面仕様
- 日時は `YYYY-MM-DD HH:mm` 形式で表示する
- 改行は表示時に保持する
- 入力エラーは対象項目ごとに表示する
- 通信失敗時は画面上部またはフォーム直下に再試行可能なメッセージを表示する

## 共通 API 方針
- API は JSON を返す
- 正常系は 2xx、入力不備は 4xx、想定外エラーは 5xx を返す
- 削除済み投稿/コメントは API から返却しない
- 投稿作成、更新、削除、コメント作成の結果は、再読込時にも同じ整合状態を返す
- バリデーションエラー時は、少なくとも `message` と項目単位の `fieldErrors` を返せる構造を持つ
- `404` `403` `500` でも JSON の基本構造を維持する

## エラーハンドリング
- 存在しない投稿 ID 指定時は `404 Not Found`
- 入力不備時は `400 Bad Request`
- 編集用キー不一致時は `403 Forbidden`
- 成功メッセージと失敗メッセージは同一操作結果として同時に表示しない
- 削除済み投稿への詳細表示、更新、削除、コメント作成は `404 Not Found` に統一する
- 一覧 API の不正な `page` または `limit` は `400 Bad Request` とする

## セキュリティ・運用前提
- HTML タグは文字列として扱い、表示時はエスケープする
- 連続投稿対策は MVP では簡易でよいが、最低限のサーバー側バリデーションを行う
- 編集用キーは平文保存を避け、ハッシュ化保存を必須とする
- コメント削除機能は MVP スコープ外とし、`deletedAt` は将来拡張の予約フィールドとして扱う
