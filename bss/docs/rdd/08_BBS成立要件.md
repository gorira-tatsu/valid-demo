# 08. BBS 成立要件

## 目的
- MVP の掲示板として最低限成立していると判断するための横断要件を定義する。
- 個別機能の要件だけでなく、投稿ライフサイクル全体の整合を確認対象に含める。

## 成立条件
- 利用者は一覧から最新の未削除投稿を見つけ、詳細へ遷移できる。
- 利用者は新規投稿を作成し、成功後に自分の投稿を詳細で確認できる。
- 利用者は詳細画面でコメントを読み、同一画面からコメントを投稿できる。
- 投稿者本人は編集用キーで投稿を更新または削除できる。
- 削除済み投稿は一覧、詳細、コメント投稿の各導線から一貫して除外される。
- 初見の利用者でも、必須項目表示、入力エラー、一覧へ戻る導線、編集用キー保管の注意により操作完了まで到達できる。

## 横断業務ルール
- 投稿作成成功後は一覧または詳細で新規投稿が可視になる。
- 投稿更新成功後は詳細で更新内容を確認でき、更新日時が反映される。
- 投稿削除成功後は一覧、詳細、コメント投稿 API のすべてで再利用不可とする。
- コメント投稿成功後は同一詳細画面で件数と内容が即時に反映される。
- 一覧のコメント件数と詳細で閲覧できる未削除コメント件数は矛盾しない。
- 一覧再取得や詳細再取得を行っても、同一データ集合に対する並び順は安定している。
- エラー応答は `message` と必要に応じて `fieldErrors` を持つ JSON として一貫して扱える。

## 失敗時の成立条件
- 入力不備では保存せず、入力内容を保持して修正可能である。
- サーバーエラーでは成功メッセージと失敗メッセージを同時表示しない。
- 存在しない投稿、削除済み投稿への操作は `404 Not Found` または同等の非表示動作に統一する。
- 編集用キー不一致では更新も削除も成立せず、再入力可能な状態を保つ。
- 投稿作成とコメント投稿では、送信中の重複送信を成立させない。
- 再試行で回復した後は stale な失敗表示を残さず、通常操作に戻れる。
- 更新済み投稿の詳細取得が一時的に失敗しても、再試行で回復した後は更新日時表示とコメント件数整合が維持される。
- 一覧 API の不正な `page` `limit`、存在しない投稿 ID、削除済み投稿への更新/削除/コメント作成は、実装全体で矛盾なく失敗する。

## 今回の `valid` 対応
- 投稿作成から一覧、詳細、更新反映、コメント件数整合、削除後コメント拒否までの横断整合:
  - `BoardFlowModel`
- 一覧の未削除投稿表示、詳細導線、ページング:
  - `PostListModel`
  - `ListRenderingModel`
- 詳細でのコメント空状態、更新日時表示、削除済み 404:
  - `PostDetailModel`
- 編集用キー一致、不一致、削除確認、削除後非表示:
  - `EditDeleteModel`
- コメント成功反映、失敗時入力保持、削除済み投稿拒否:
  - `CommentModel`
- 成功メッセージと retry banner の排他、エラー時導線:
  - `PresentationContractModel`
  - `RetryUxModel`
- 作成/コメントの二重送信防止と失敗後再送規律:
  - `SubmissionDisciplineModel`
- 更新済み投稿の詳細取得失敗から retry 回復後に、`updatedAt` 表示と一覧/詳細のコメント件数整合が復元されること:
  - `BoardFlowModel`
